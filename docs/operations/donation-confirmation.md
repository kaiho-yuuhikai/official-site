# 振込確認 運用フロー手順書

寄付申し出フォームから入った申し出に対し、運営担当が銀行通帳の入金を確認してスプレッドシートのステータスを更新するまでの手順をまとめる。

このフローが滞ると、HP の「累計寄付額」「支援者数」表示が実態と乖離し、寄付者への謝意も遅れるため、定常運用として確実に回す必要がある。

---

## 1. 役割

| 役割 | 担当 | 主な作業 |
|---|---|---|
| **主担当** | 会計担当（運営役員から1名） | 通帳照合、ステータス更新、メール対応 |
| **副担当** | 別の運営役員から1名 | 主担当が不在時の代行（最低週1回はチェック） |
| **エスカレーション先** | 同窓会会長 | 不審な申し出、判断に迷うケース |

主担当・副担当は新体制発足時に役員会で決定し、本ドキュメントの末尾「担当履歴」に記録する。

## 2. チェック頻度

- **主担当**: 週1回（推奨曜日: 月曜 or 火曜）
- **副担当**: 主担当不在週のみ。月初に「今月の主担当不在週」を共有
- **月次**: 月末に主担当が「未確認」のまま2週間以上経過した行を一覧化し、申し出者に問い合わせメールを送る

## 3. 振込確認の判断基準

入金1件に対し、対応する申し出行を以下の優先順で照合する。

1. **氏名一致**（フリガナ含む）
2. **金額一致**
3. **入金日が申し出から30日以内**

3つすべて一致すれば「確認済」とする。

### グレーゾーン

| 状況 | 判断 |
|---|---|
| 氏名は完全一致だが金額が違う | 申し出者にメールで照会。金額確定後に「確認済」、または金額を実額に修正 |
| 申し出が「その他金額」で曖昧 | 入金額をF列「その他金額」に記入し、I列「確認済」 |
| 同姓同名で複数の申し出がある | 入学期 + 連絡先メールで識別。判別不能なら申し出者へ照会 |
| 1人が複数口を分割振込 | 各口を別行とし、それぞれ「確認済」 |
| 1人が複数口を一括振込 | 申し出行を **行追加 or 申し出金額の修正** で一致させる（主担当判断） |
| 申し出の身に覚えがないと振込元から連絡 | 「キャンセル」へ変更、メールで返金対応 |
| 申し出から30日以上入金なし | 月次ジョブで申し出者にリマインダ、さらに30日経過で「キャンセル」化 |

## 4. スプレッドシート更新手順

1. スプレッドシートを開く（URLは運営内部の共有ドキュメントを参照）
2. 振込確認できた行を見つける
3. **I列「振込確認ステータス」**: プルダウンから「確認済」を選択
4. **J列「確認日」**: 入金日を入力（YYYY-MM-DD 形式）
5. **K列「確認者」**: 自分の氏名（または役職名）を入力
6. 必要に応じて H列「メッセージ」に運営側メモを追記（例: 「2口分一括振込」）

> ⚠️ 一度「確認済」にした行を後から修正する場合は、必ず Slack/LINE WORKS など別チャネルで運営内部に共有し、誰がいつ何を変えたか記録を残す。

## 5. メール対応

寄付申し出フォーム送信時に、振込先案内の自動返信メールが送られる（フォーム送信トリガー）。

主担当の追加対応:

- **入金確認後**: 確認御礼のメールを返信（任意。テンプレートは Drive 内の `donation-thanks-template.md`）
- **30日リマインダ**: 自動でなくBcc手動送信。テンプレートあり
- **キャンセル化時**: 申し出者に経緯説明メール

## 6. 月次照合

月末（毎月最終営業日）に主担当が以下を実施:

- [ ] 通帳の入金一覧と「確認済」行の合計が一致するか
- [ ] 「未確認」のまま2週間以上経過した行を抽出 → リマインダメール
- [ ] 「確認済」行のうち、HP（公式サイト）の累計額と乖離がないか確認
- [ ] 役員会向けに「今月の寄付件数 / 累計額」を1行で共有

HPの累計額は GitHub Actions が日次で更新。月次照合時に乖離していれば、Actions ワークフロー (`deploy.yml`) を手動トリガーして即時反映できる。

## 7. エスカレーション

以下の場合は会長に即時相談する:

- 1件あたり 100,000円 を超える申し出
- 寄付者の身元・連絡先が不明確
- 申し出後に振込元から「身に覚えがない」旨の連絡があった場合
- 「キャンセル」化を判断した行で、すでに入金が口座に着金している場合
- フォームに不適切な内容（誹謗中傷、プライバシー侵害等）が含まれていた場合
