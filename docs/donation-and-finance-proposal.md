## 1. 背景と課題

### 1-1. 現状の課題（屋良さんの整理より）

- 入出金の件数自体は少ないが、**誰が立替・預かりしているかの把握**が困難
- イベント実施後の収支連携が遅れがち（数年分の精算が滞留した実績あり）
- 事業の計画→承認→実施→報告→精算の共通フローが確立されていない
- 大同窓会のような単発大型収入に依存する運営は負担が大きい

### 1-2. 目指す姿

- 寄付金の常設化とHPでのリアルタイム見える化
- 事業ごとの申請→承認→報告→精算フローの標準化
- 会計情報の透明性確保（HP上での収支報告公開）

## 3. Phase 1: 寄付金システム

### 3-1. 全体フロー

```
寄付者がHPの「寄付する」ボタンをクリック
  ↓
Googleフォームで寄付申し出を入力
  （氏名、期、金額、寄付先基金の選択、メッセージ）
  ↓
Googleスプレッドシートに自動記録
  ↓
会計担当が振込確認後、スプレッドシートのステータスを「確認済」に更新
  ↓
GitHub Actionsが定期的にスプレッドシートからデータ取得
  ↓
HPの寄付セクションに自動反映（累計金額、支援者数、進捗バー）
```

### 3-2. 決済方法

現時点では **銀行振込** をベースとする。

- フォーム送信後、振込先口座情報を自動返信メールで案内
- 将来的にオンライン決済（Stripe等）を導入する場合は別途検討

**理由**:
- 銀行口座連携は月額手数料が発生する（屋良さん指摘）
- 寄付の頻度を考えると、まずは振込ベースで十分
- オンライン決済導入は寄付件数が増えてからでも遅くない

### 3-3. Googleフォーム設計（寄付申し出）

| 項目 | 種類 | 必須 | 備考 |
|------|------|------|------|
| お名前 | テキスト | ○ | |
| 卒業期 | プルダウン | ○ | 1期〜37期 + 関係者 |
| メールアドレス | テキスト | ○ | 振込先案内の送付先 |
| 寄付先基金 | ラジオボタン | ○ | 探究活動基金 / 芸術家応援基金 / 一般寄付 |
| 寄付金額 | プルダウン or テキスト | ○ | 3,000円 / 5,000円 / 10,000円 / その他 |
| メッセージ（任意） | テキストエリア | | HP掲載可否のチェックボックス付き |
| HP掲載の可否 | ラジオボタン | ○ | 氏名掲載OK / 匿名希望 |

| J | **確認日** | 会計が振込確認した日付 |
| K | **確認者** | 確認した会計担当者名 |

### 3-5. HP反映の仕組み

既存のnote RSS取得と同じ技術パターンで実装する。

```
GitHub Actions（1日1回 or 手動トリガー）
  ↓
Google Sheets API でスプレッドシートの「確認済」データを取得
  ↓
JSON ファイルとして public/data/donations.json に出力
  ↓
Nuxt の静的サイト生成時にデータを読み込み
  ↓
寄付セクションに反映:
  - 累計寄付金額（確認済のみ合算）
  - 支援者数
  - 基金別の進捗バー
  - 寄付者一覧（掲載OKの方のみ）
```

### 3-6. HP表示イメージ

現在サイトに表示中のモックデータ:
- 探究活動基金: ¥390,000 / ¥500,000（78%）
- 芸術家応援基金: ¥180,000 / ¥300,000（60%）
- 累計寄付: ¥1,250,000（87名）

→ これをスプレッドシートの実データに差し替える

## 4. Phase 2: 事業精算フロー

### 4-1. 全体フロー

```
① 事業計画の申請（Googleフォーム）
  ↓
② 承認会議体で承認
  ↓
③ 事業実施
  ↓
④ 実施報告・収支報告の入力（同じフォームの続き）
  ↓
⑤ 会計に連携 → 立替者・預かり者との精算
  ↓
⑥ 精算完了記録
```

### 4-2. フォーム設計方針

屋良さんの要望: **計画（01）と報告（02）を1つのフォームにまとめたい**

→ Googleフォームの **セクション機能** で実現可能

```
セクション1: 基本情報（事業名、担当者、実施予定日）
  ↓
セクション2: 計画（予算見込み — 収入項目・支出項目）
  ↓
[承認後、同じレコードに追記]
  ↓
セクション3: 実施報告（実施概要、参加者数）
  ↓
セクション4: 収支実績（実際の収入・支出、立替者情報、預かり者情報）
```

**注意**: Googleフォームは1回の送信で完結するため、「計画時に送信→報告時に追記」は標準機能では難しい。以下の代替案を検討:

**案A: 2段階フォーム方式**
- フォーム①: 事業計画申請（計画時に送信）
- フォーム②: 事業報告・精算（実施後に送信、①の事業IDを入力）
- スプレッドシート上でIDで紐づけ

**案B: スプレッドシート直接入力方式**
- 計画段階でフォームから入力
- 報告段階ではスプレッドシートに直接追記（権限管理で対応）

**推奨: 案A**（フォーム入力の方が入力ミスが少なく、スマホからも操作しやすいため）

### 4-3. 支出項目（屋良さん・上間さんの議論より）

| カテゴリ | 例 |
|---------|-----|
| 会場費 | 会議室使用料、懇親会会場費 |
| 講師関連 | 謝礼、交通費、宿泊費 |
| 備品・消耗品 | 資料印刷、文具 |
| 飲食 | 懇親会飲食代、弁当代 |
| その他 | 通信費、振込手数料 |

### 4-4. 重要ポイント（屋良さんの指摘）

- 事業採算制ではないので、収支差額のマイナスに対する「補填方針」は不要
- **収入合計と支出合計の絶対値の比較**を承認基準として重視
- 事業部門ごとに一時的な会計役割を設ける組織案と連動

## 6. 技術構成まとめ

### 使用ツール（すべて無料）

| ツール | 用途 |
|--------|------|
| Google Forms | 寄付申し出、事業計画・報告の入力 |
| Google Sheets | データの蓄積・管理・集計 |
| GitHub Actions | スプレッドシート → HP への自動データ連携 |
| Nuxt 3（既存サイト） | 寄付セクション・収支報告ページの表示 |

### アーキテクチャ図

```
[寄付者 / 事業担当者]
        |
        v
  [Google Forms]
        |
        v
  [Google Sheets] ←── 会計担当が確認・更新
        |
        v (GitHub Actions: 定期取得)
  [public/data/donations.json]
        |
        v
  [HP（Nuxt 3 静的サイト）] ←── GitHub Pages で公開
```

### 既存の類似実装（参考）

note RSS 取得の仕組みが同じパターン:
- GitHub Actions で外部データ（note RSS）を定期取得
- JSON ファイルとしてリポジトリに保存
- 静的サイト生成時に読み込んで表示

## 7. Google Workspace アカウントについて

上間さんが開邦雄飛会のGoogle Workspaceアカウント作成を提案。

### 推奨構成

- **ドメイン**: 既存ドメインがあれば活用（なければ無料のGoogleアカウントでも可）
- **共有ドライブ**: 各種書類のストック先
- **共有スプレッドシート**: 寄付管理・事業精算データ
- **共有フォーム**: 寄付申し出・事業計画/報告

### 注意点

- Google Workspace（有料版）は月額費用が発生する
- まずは **個人Googleアカウント（無料）** を同窓会共有アカウントとして作成し、Google Drive / Sheets / Forms を運用する方が現実的
- 将来的に独自ドメインのメールアドレスが必要になった段階でWorkspace移行を検討

## 付録: 上間さんが作成済みのレシート読み取り自動化について

上間さんが当初提案した「レシート撮影→Gemini API→スプレッドシート→HP反映」の自動化フローは技術的には実現可能ですが、以下の理由から現時点では保留とします:

- 出金の件数が少ない（屋良さん指摘）
- 立替者・預かり者の把握が先決で、レシート読み取りはその後の話
- 情報連携のフロー（Phase 2）が確立された後に、必要に応じて導入を検討

また、上間さんが作成した収支報告アプリ（https://project-finance-blue.vercel.app/）は、将来的にPhase 3の収支報告ページの参考UIとして活用できる可能性があります。
