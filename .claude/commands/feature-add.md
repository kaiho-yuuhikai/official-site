# 機能追加・改修の要望を Issue 化

「こうしたい」「こう変えたい」というアイデアを Claude が整理して GitHub Issue を作成します。
Issue 化することで `/feature-implement` で実装に進めるようになります。

このコマンドは **ビジネスユーザー (上間さん等) でも使える** よう、専門用語を避けて対話的に進めます。

## 手順

### Step 0: 要件のキャッチアップ

ユーザーの要件: $ARGUMENTS

引数で要件が渡されていればそれを起点に、なければユーザーに最初の質問を投げる:

> 「どんな変更をしたいですか? 例えば『フォームに紹介者欄を追加したい』『寄付者一覧を期ごとに分けて表示したい』のように、ざっくりとで構いません。」

### Step 1: 必要な情報のヒアリング

以下が揃うまで質問を続ける (一度に多く聞かない、1〜2 問ずつ):

| 項目 | 例 |
|---|---|
| **何を変えたい?** | 「寄付フォームに『紹介者』項目を追加」 |
| **どこに表示する?** | 「フォームと、HP の寄付者一覧」 |
| **誰のため?** | 「同窓生の紹介で寄付が増えたケースを記録したい (会計の屋良さん要望)」 |
| **緊急度** | 「次回振込確認まで (5/末)」 / 「いつでも」 |
| **画像・参考資料** | 既存ページのスクショ / 他団体サイトの URL 等 |

すべて埋まらなくても OK。「わからない」場合は『相談で決める』として進める。

### Step 2: Issue 内容の整形

ヒアリングした内容から、以下の構造で Issue body を組み立てる:

```markdown
## 背景・目的

(誰の何の困りごとを解決したいか、1〜3 文)

## やりたいこと

- 箇条書きで具体的に
- 「〇〇画面に□□を追加」
- 「△△の動作を××に変える」

## Acceptance Criteria (受入条件)

- [ ] (検証可能な条件 1)
- [ ] (検証可能な条件 2)
- [ ] (テストが必要なら明記)

## 関連

- 関連 Issue / PR (もしあれば)
- 参考リンク (もしあれば)

## 緊急度

(いつまでに欲しいか / 任意)
```

### Step 3: ユーザー確認

組み立てた Issue 内容をユーザーに見せて確認:

```
作成する Issue はこちらでよいですか?

タイトル: feat: 〇〇を追加

(本文)

「OK」「修正したい」のどちらかで答えてください。
```

修正指示があれば Step 2 に戻ってループ。

### Step 4: Issue 作成

リモート URL から対応する GitHub アカウントを特定:

```bash
git remote -v
```

CLAUDE.md グローバル設定の対応表に基づき `GH_TOKEN` を設定:

```bash
GH_TOKEN=$(gh auth token --user <対応アカウント>) gh issue create \
  --title "<整形したタイトル>" \
  --body-file /tmp/feature_issue_body.txt
```

Issue 本文は **必ず /tmp 経由のファイル**で渡す (heredoc は括弧で壊れる)。

### Step 5: 完了報告

```
✅ Issue #N を作成しました
   <Issue タイトル>
   https://github.com/<org>/<repo>/issues/N

次のステップ:
  - 詳細を後で確認・編集したい場合: Issue ページで直接編集
  - すぐ実装に進める場合: /feature-implement N
```

## 注意事項

- **Windows でも動作**: `gh` / `git` / Node.js コマンドはクロスプラットフォーム
- **秘匿情報を Issue に書かない**: トークン・パスワード・個人連絡先は別途共有 (Slack DM / 1Password)
- **要件が大きすぎる場合**: 「Phase 分割を提案」と案内し、まず最小 1 Phase 分の Issue を作る (例: Phase 1 = フォーム改修、Phase 2 = HP 表示、Phase 3 = 通知)
- **既存 Issue と重複しないか確認**: `gh issue list --search "<キーワード>"` で重複チェック

## 用語の言い換え (ユーザー向け表示)

- Issue → 「やりたいことのチケット」 / 「課題」
- AC (Acceptance Criteria) → 「完了の条件」
- Pull Request (PR) → 「変更案」
- Implementation → 「実装」 / 「組み込み」
