# サイト公開（GitHub Pages へデプロイ）

ローカルの変更を自動テスト通過後に GitHub Pages に反映します。
他のメンバーの変更との競合を自動で検出・防止します。

## 手順

### Step 1: 自動テスト実行（ゲートチェック）

公開前に全テスト（ビルドテスト含む）を実行:

```bash
npx playwright test 2>&1
```

**テスト結果に応じた対応（ユーザーに透過的に実行）:**

- **全テスト通過**: Step 2 に進む
- **テスト失敗**:
  1. エラーを分析し、コードを自動修正
  2. 再テスト（最大3回ループ）
  3. 全テスト通過したら Step 2 へ
  4. 修復できない場合はユーザーに報告し、公開を中止

**重要: テストが全て通過しない限り、絶対にコミット・プッシュしないこと。**

### Step 2: 他のメンバーの変更を取り込み（衝突防止）

**重要: 公開直前に必ず最新の状態を取り込む。これにより他のメンバーの変更を上書きすることを防ぐ。**

ローカルの変更を一時退避してから最新を取得:

```bash
git stash
git pull --rebase origin main 2>&1
git stash pop 2>&1
```

（変更がない場合は stash なしで直接 pull する）

**競合が発生した場合:**
1. `git rebase --abort && git stash pop` で元の状態に戻す
2. ユーザーに以下を伝える:
   「他のメンバーが同じ箇所を最近変更したため、自動で統合できませんでした。
   どちらの変更を優先するか確認が必要です。変更内容を教えてください。」
3. **公開を中止する**（データを失わないために、競合時は絶対に強制的に公開しない）

**競合なしで同期完了した場合:**
他のメンバーの変更を取り込んだ後、テストを再実行して問題がないことを確認:

```bash
npx playwright test 2>&1
```

テストが失敗した場合は自動修復を試み（最大3回）、修復できない場合は公開を中止してユーザーに報告する。

**ユーザーへの表示**: 「他のメンバーの変更を確認しています...」のみ。

### Step 3: 変更内容の確認

```bash
git status
git diff
```

変更内容の概要をユーザーに日本語で分かりやすく説明してください。
技術的な詳細ではなく、「何がどう変わるか」をビジネスの観点で伝えてください。
テスト結果も「全ての品質チェックに合格しました」と簡潔に伝えてください。

### Step 4: ユーザーに最終確認

以下を確認してください:
- ローカルプレビューで変更内容を確認済みですか?
- この内容で公開サイトに反映して問題ありませんか?

ユーザーが「OK」「はい」「問題ない」等の肯定的な回答をした場合のみ、次のステップに進んでください。

### Step 5: 変更の保存と公開

変更内容に基づいて適切なコミットメッセージを日本語で作成し、保存・公開する。

```bash
git add -A
git commit -m "適切なコミットメッセージ"
```

コミットメッセージの規約:
- 新機能追加: `feat: 〇〇を追加`
- 既存機能の更新: `update: 〇〇を更新`
- テキスト・コンテンツ修正: `content: 〇〇を修正`
- デザイン変更: `style: 〇〇のデザインを変更`
- バグ修正: `fix: 〇〇を修正`

公開（プッシュ）:

```bash
git push origin main 2>&1
```

**プッシュが拒否された場合（他のメンバーが同時に公開した場合）:**
1. 再度最新を取り込む:
   ```bash
   git pull --rebase origin main 2>&1
   ```
2. 競合がなければ再度プッシュ
3. 競合がある場合は `git rebase --abort` で中止し、ユーザーに報告:
   「他のメンバーがちょうど同じタイミングで公開したため、変更の統合が必要です。もう一度 `/site-publish` を実行してください。」

**ユーザーへの表示**: 「公開サイトに反映しています...」のみ。

### Step 6: デプロイ確認

プッシュ後、GitHub Actions の実行状況を確認してください。

```bash
gh run list --limit 1
```

ユーザーに以下を案内:
- GitHub Actions でビルド・デプロイが自動実行されます（約2-3分）
- デプロイ完了後、公開サイトに反映されます

## 注意事項

- `main` ブランチへの push で自動デプロイが実行されます
- テスト未通過のコードは絶対にプッシュしないこと
- 競合が発生した場合は絶対に強制的に公開しないこと（データ損失防止）
- ユーザーにはGit用語（commit, push, pull, rebase, stash等）を使わず、ビジネス用語で説明すること
  - commit → 「変更を保存」
  - push → 「公開サイトに反映」
  - pull → 「最新の状態を取得」
  - conflict → 「他のメンバーの変更との競合」
- デプロイに失敗した場合は GitHub Actions のログを確認してください
